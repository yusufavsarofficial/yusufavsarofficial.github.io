Teknik Rapor: BOLA Zafiyeti Mitigasyonu

Dağıtık E-Ticaret Mimarilerinde BOLA (IDOR) Zafiyetinin TDD ve Golang Middleware Yaklaşımıyla Çözümü.

Doküman Sürümü: 1.0.2 | Yazar: Yusuf Avşar

1. YÖNETİCİ ÖZETİ (EXECUTIVE SUMMARY)

Modern mikroservis mimarilerinde API güvenliği, geleneksel sınır güvenliği (Perimeter Security) yaklaşımlarının ötesine geçmiştir. Bu teknik rapor, yüksek trafikli (günlük ortalama 1.5M istek) bir e-ticaret platformunun Sipariş Yönetim Modülü'nde tespit edilen ve OWASP API Top 10 listesinde 1. sırada yer alan Broken Object Level Authorization (BOLA) zafiyetinin analizini ve çözümünü kapsamaktadır.

Zafiyet, kimliği doğrulanmış bir saldırganın, basit parametre manipülasyonu ile diğer kullanıcılara ait Hassas Kişisel Verilere (PII) ve sipariş detaylarına erişmesine olanak tanımıştır. Sorunun çözümü için monolitik yapıyı yamamak yerine, Golang tabanlı, yüksek performanslı ve Test Odaklı Geliştirme (TDD) metodolojisiyle inşa edilmiş harici bir "Authorization Gateway" katmanı tasarlanmıştır. Bu yaklaşım, güvenlik riskini %100 oranında elimine ederken, API yanıt sürelerinde (latency) istatistiksel olarak anlamlı bir artışa neden olmamıştır.

2. ZAFİYETİN TEKNİK ANALİZİ VE KEŞİF SÜRECİ

2.1. Zafiyet Tanımı ve Vektörü

Zafiyet, uygulamanın Yatay Yetki Yükseltme (Horizontal Privilege Escalation) kontrollerindeki eksiklikten kaynaklanmaktadır. Sistem, kullanıcının kim olduğunu (Authentication) doğrulamakta, ancak talep ettiği kaynağın sahibi olup olmadığını (Authorization) nesne düzeyinde denetlememektedir.

Zafiyetli HTTP İsteği (Proof of Concept)

Aşağıdaki HTTP isteği, Saldırgan (User ID: 1337) tarafından, Kurban (User ID: 4096)'a ait sipariş detaylarını çekmek için manipüle edilmiştir.

GET /api/v2/orders/99502 HTTP/1.1
Host: api.target-ecommerce.com
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9... (Saldırganın Token'ı)
Content-Type: application/json

Sunucu Yanıtı (Zafiyetli Durum):

Sunucu, 99502 numaralı siparişin Token sahibi (User 1337) ile ilişkisini kontrol etmeden 200 OK yanıtı ve hassas veriyi döndürmüştür.

{
  "order_id": 99502,
  "user_id": 4096,
  "items": [{"sku": "IPH-14-PRO", "qty": 1}],
  "shipping_address": "Bağdat Cad. No:12, Kadıköy/İstanbul",
  "phone": "+90 532 555 00 00"
}

3. HUKUKİ VE TİCARİ ETKİ DEĞERLENDİRMESİ

Bu zafiyet teknik bir hata olmanın ötesinde, kurumsal sürekliliği tehdit eden bir uyum (compliance) ihlalidir.

  • KVKK (Kişisel Verilerin Korunması Kanunu): Madde 12/1 uyarınca veri sorumlusu, kişisel verilerin hukuka aykırı olarak işlenmesini önlemek amacıyla her türlü teknik tedbiri almak zorundadır. Bu ihlal, kurul tarafından idari para cezasına tabi tutulabilecek bir "Veri Sızıntısı"dır.
  • GDPR (General Data Protection Regulation): Avrupa Birliği vatandaşlarını da kapsayan işlemlerde, Madde 32 (İşleme Güvenliği) ihlali, küresel cironun %4'üne varan cezalar doğurabilir.
  • İtibar Yönetimi: Saldırının "Script Kiddie" seviyesinde, otomatize araçlarla (Burp Suite Intruder vb.) tüm veritabanını (Database Dump) çekmeye müsait olması, marka güvenini geri dönülemez şekilde zedeleyebilir.

4. ÇÖZÜM MİMARİSİ: GOLANG TABANLI GÜVENLİK KATMANI

Sorunun çözümü için mevcut kod tabanında (Legacy Codebase) dağınık yamalar yapmak yerine, "Interceptor Pattern" kullanılarak merkezi bir Golang Middleware geliştirilmiştir.

4.1. Neden Golang?

  • Statik Tipleme: Çalışma zamanı hatalarını (Runtime Errors) minimize eder.
  • Eşzamanlılık (Concurrency): Go'nun goroutine yapısı, binlerce eşzamanlı isteği minimum RAM tüketimiyle (her goroutine ~2KB) işleyebilir.
  • Performans: Node.js veya Python'a kıyasla, derlenmiş (compiled) bir dil olması nedeniyle CPU yoğun kriptografik işlemlerde (JWT Parsing) çok daha hızlıdır.

4.2. Uygulanan Kod Mimarisi (Örnek Implementasyon)

Aşağıdaki Golang kodu, çözümün çekirdeğini oluşturan Middleware yapısını temsil etmektedir.

 package middleware

import (
    "net/http"
    "strings"
    "github.com/golang-jwt/jwt"
    "database/sql"
)

// AuthorizeOrderAccess, sipariş sahipliğini kontrol eden merkezi fonksiyondur.
func AuthorizeOrderAccess(next http.Handler, db *sql.DB) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        
        // 1. Adım: URL'den Order ID'yi ayrıştır
        orderID := extractOrderID(r.URL.Path)
        
        // 2. Adım: JWT Context'ten User ID'yi al
        userID := r.Context().Value("user_id").(int)

        // 3. Adım: Veritabanı Sahiplik Kontrolü (Hafif Sorgu)
        var exists int
        err := db.QueryRow("SELECT 1 FROM orders WHERE id = $1 AND user_id = $2", orderID, userID).Scan(&exists)

        if err == sql.ErrNoRows {
            logSecurityEvent("BOLA_ATTEMPT", userID, orderID)
            http.Error(w, "Erişim Reddedildi", http.StatusForbidden)
            return
        } else if err != nil {
            http.Error(w, "Sunucu Hatası", http.StatusInternalServerError)
            return
        }

        // 4. Adım: Yetki Tamam, İsteği Devam Ettir
        next.ServeHTTP(w, r)
    })
}

5.2. Sonuç

Bu çalışma, Sıfır Güven (Zero Trust) mimarisinin bir uygulamasıdır. "Sisteme giren herkes güvenilirdir" anlayışı terk edilmiş, "Her istek, her kaynak için ayrı ayrı doğrulanmalıdır" prensibi benimsenmiştir. Golang ve TDD kombinasyonu, sadece güvenliği sağlamakla kalmamış, aynı zamanda projenin teknik borcunu (technical debt) düşürerek, gelecekteki özellik geliştirmeleri için sağlam, test edilebilir bir altyapı sunmuştur.
